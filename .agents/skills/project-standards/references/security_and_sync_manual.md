# Security & Synchronization Manual

This manual describes the mechanisms designed to ensure player data integrity, prevent cheating, and manage real-time concurrency.

---

## 🛡️ Anti-Cheat & Sanitization Logic

The security system is based on continuous validation of the game state before every persistence event.

### 1. Data Validation (`validateAndSanitize`)

Every time the game attempts to save, the data passes through an integrity filter that:

* **Corrects Numerical Values**: Ensures Money, BattleCoins, and levels are not negative.
* **Inventory Cleanup**: Removes negative quantities in items.
* **UID Integrity (Anti-Cloning)**: The system verifies that no two Pokémon share the same **Unique ID (UID)** across the team and box. Duplicates are automatically removed to prevent cloning exploits.

### 2. Security Versions (v1 vs v2)

* **v1 Accounts (Legacy)**: Receive "silent" sanitization. The system cleans the data and saves it. Upon the first successful save, the account is automatically migrated to v2.
* **v2+ Accounts (Strict)**: If a critical inconsistency is detected (such as duplicate UIDs), the system triggers the **Rollback Protocol** instead of attempting to sanitize and save potentially corrupt data.

### 3. Combat Shielding (Stat Freezing)

To avoid cache manipulation during a fight (e.g., hacking money via console):

* **Battle Snapshot**: Upon starting a fight (`START_BATTLE`), the system takes a snapshot of money, XP, and critical items.
* **Delta Validation**: Upon finishing, the system only accepts the increment generated by the battle engine. If the values exceed logical sanity limits, the save is rejected.

## 🔄 Rollback System

There are two types of Rollback designed to protect the user's save file:

### 1. Local Inconsistency Rollback

If the engine detects that the current state in memory is invalid (e.g., duplicate UIDs in a v2 account), the save is canceled, and the game downloads the last known valid copy from the Cloud server.

### 2. Concurrency Rollback (`OUT_OF_SYNC`)

To prevent an old tab from overwriting progress from a new tab, the system uses an **Expected ID**:

* Each save generates a unique ID on the server.
* When attempting to save, the client sends the ID it believes is current (`p_expected_id`).
* If the server has a different ID, it means the user saved from another device or tab. The server rejects the operation, and the client performs a rollback to the server's version.

### 3. Post-Combat Synchronization (Delta Merge)

If you receive an external update (such as an accepted trade) while in battle:

* **Deferred Synchronization**: The system queues external changes to avoid corrupting the active fight.
* **Result Merge**: After the battle, if a de-sync conflict occurs, the client downloads the new "truth" from the server and applies the results (EXP, Gold) earned in the local combat (**Delta Merge**).

## ⏱️ Cache & Throttling (60-Second Sync)

To minimize server load and optimize performance on mobile devices, the project applies the **60-Second Synchronization Principle**.

* **Resource Savings**: Minor changes are not sent instantly to the cloud. They accumulate in a local cache that syncs periodically.
* **Critical Events**: Certain actions (winning a badge, catching a legendary, performing a trade) force an immediate save, ignoring the 60-second interval.
* **Cache Invalidation**: Upon login or opening a new tab, the system **MUST** invalidate any local cache and force a complete cloud download.
  * **Technical Action**: Execute `localStorage.removeItem('save_throttle_cache')` to remove pending sync data and ensure the starting point is the absolute "Source of Truth".
* **Pre-Action Flush Protocol**: Before any social action other than chat (sending a trade, GTS listing, etc.), the system triggers a forced atomic save to ensure synchronization integrity.
* **Mandatory Claim Sync**: When using the "Receive All" function or claiming an individual asset, the system forces a prior save to prevent the local state from overwriting the incoming data.

## 📡 Websockets & Realtime

The project uses **Supabase Realtime** (Websockets) to keep the world and session states synced without needing a page reload.

### 1. Single Session Lock (`session_lock`)

The DBRouter subscribes to a specific channel for the user. If a database change made by another instance (another tab) is detected, the channel emits an event that:

* Displays a locking overlay.
* Automatically closes the local DBRouter's write permissions.

### 2. Social Systems & PvP

The Realtime system is the engine behind:

* **Global Chat**: Distributed instant messaging.
* **GTS (Global Trade System)**: Real-time updates for item and Pokémon listings and sales.
* **Live PvP**: Invitation management and battle states through event broadcasting.

## 💎 Asset Management & Abuse Control

Server-centric architecture to ensure the integrity of Pokémon and items and prevent transaction spam.

### 1. Server-Side Escrow Model

Every trade or sale moves assets out of the player's inventory and places them under direct server custody.

* **Claim Queues**: Received (or canceled) assets are not automatically re-injected into the save. They are held in a "Pending Claim" table.
* **Active Claiming**: The user must click "Receive" to download the asset to their inventory. This process is atomic and requires a prior save from the local cache to the cloud.
* **'Receive All' Function**: Batches all pending assets. In case of delays, the system displays a progress Toast (e.g., "Processing 3/10 assets...") to inform the user.

### 2. Anti-Spam Policies & Throttling

To prevent script abuse and database overload, the following limits are applied:

* **Interface Throttling**: Mandatory **5-second cooldown** between each individual claim action. If clicked earlier, the system displays a Toast: *"Wait 5 seconds before claiming again"*.
* **Modular Quotas (Centralized Config)**: Each interaction type has an independent capacity limit of **50 slots**, easily adjustable from the global constants file:
  * Maximum 50 pending Friend Requests.
  * Maximum 50 active Trade Offers.
  * Maximum 50 simultaneous GTS Listings.

### 3. Orphan Record Cleanup

To keep the database optimized and prevent stale data accumulation:

* **Automatic Deletion**: If both players involved in a trade (sender and recipient) cease to exist or their accounts are deleted, the system must automatically purge those records.
* **Referential Integrity**: Priority is given to foreign keys with cascade deletion (`ON DELETE CASCADE`) or scheduled maintenance processes to clean up offers that can no longer be claimed by either party.
